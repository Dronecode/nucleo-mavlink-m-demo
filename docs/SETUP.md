# Setup

Full bring-up: toolchains, the MAVLink dialect, both boards, and the PX4-side
config. Do topology A first (Nucleo alone) to prove the sketch, then add PX4 for
topology B.

Paths and ports below are the ones used on the bench Mac. Adjust as needed:

- Tropic (PX4) USB CDC: `/dev/cu.usbmodem01`
- Nucleo ST-Link VCP: `/dev/cu.usbmodem1103`
- Nucleo ST-Link flash drive: `/Volumes/NOD_F103RB`

## 1. Toolchains

### arduino-cli + STM32 core (for the Nucleo)

```sh
brew install arduino-cli

# STM32duino core
arduino-cli config init
arduino-cli config add board_manager.additional_urls \
  https://github.com/stm32duino/BoardManagerFiles/raw/main/package_stmicroelectronics_index.json
arduino-cli core update-index
arduino-cli core install STMicroelectronics:stm32
```

The Nucleo FQBN used throughout:

```
STMicroelectronics:stm32:Nucleo_64:pnum=NUCLEO_F103RB
```

### PX4 build environment (for the Tropic, topology B only)

You need a working PX4-Autopilot checkout with the standard toolchain (the
`arm-none-eabi` GCC that `Tools/setup/macos.sh` installs). If you already build
PX4 for other boards you are done. The mr-tropic target is
`nxp_mr-tropic_default`.

### Host Python

```sh
cd host
python3 -m pip install -r requirements.txt   # pymavlink>=2.4.42, pyserial
```

`pymavlink>=2.4.42` matters: older builds mishandle the shell-open retry and the
military CRC extras. Do not go below it.

## 2. Generate the MAVLink dialect

Both the C headers (firmware) and the Python dialect (host) come from
`military.xml`. That file `<include>`s `common.xml`, so a single generation
gives you both dialects. It must live in a mavlink checkout at
`message_definitions/v1.0/military.xml` — the PX4 worktree's mavlink submodule is
the canonical source here.

### Python dialect (host)

```sh
cd host
./generate_dialect.sh        # writes host/military_dialect.py
```

`generate_dialect.sh` defaults `MAVROOT` to the PX4 worktree's mavlink submodule.
Override it if your checkout is elsewhere:

```sh
MAVROOT=/path/to/PX4-Autopilot/src/modules/mavlink/mavlink ./generate_dialect.sh
```

`host/military_dialect.py` is git-ignored (it is ~2 MB of generated code).

### C headers (firmware)

The `nucleo_mavlink_m/mavlink/` tree is generated the same way, `--lang=C`:

```sh
MAVROOT=/path/to/PX4-Autopilot/src/modules/mavlink/mavlink
python3 "$MAVROOT/pymavlink/tools/mavgen.py" \
  --lang=C --wire-protocol=2.0 \
  --output=nucleo_mavlink_m/mavlink \
  "$MAVROOT/message_definitions/v1.0/military.xml"
```

These headers are checked in, so you only regenerate them if `military.xml`
changes.

## 3. Build and flash the Nucleo

```sh
FQBN="STMicroelectronics:stm32:Nucleo_64:pnum=NUCLEO_F103RB"

arduino-cli compile -b "$FQBN" nucleo_mavlink_m
arduino-cli upload  -b "$FQBN" -p /dev/cu.usbmodem1103 nucleo_mavlink_m
```

`upload` copies the compiled `.bin` onto the ST-Link mass-storage drive
(`/Volumes/NOD_F103RB`); the ST-Link flashes it from there. Footprint is about
15% flash / 9% RAM, so there is plenty of headroom on the 128 KB / 20 KB part.

Verify topology A before wiring anything:

```sh
cd host
python3 fake_pixhawk.py --arm 1
```

You should see the `HEARTBEAT`, the `ESAD_STATE` reply, and a `PASS` line.

> As shipped, the firmware puts the MAVLink link on **USART1** (for topology B)
> and uses **USART2/`Serial`** as a debug console at 115200. For a pure
> topology-A run over the ST-Link VCP, either wire a USB-serial adapter to
> USART1, or change `FCLink` in the sketch to `Serial`. See
> [WIRING.md](WIRING.md).

## 4. Build and flash the Tropic (PX4, topology B)

The general procedure — selecting the dialect, placing `military.xml`, and the
i.MXRT linker fix — is board-agnostic and documented in
**[PX4_INTEGRATION.md](PX4_INTEGRATION.md)**. Read that for the full explanation;
below is just the mr-tropic-specific instance of it.

### 4a. Dialect + build config (mr-tropic)

Set the dialect in `boards/nxp/mr-tropic/default.px4board`:

```
CONFIG_MAVLINK_DIALECT="military"
```

and make sure `military.xml` is in
`src/modules/mavlink/mavlink/message_definitions/v1.0/`. mr-tropic is an i.MXRT
board, so it **also** needs the `.ARM.exidx`/`.extab` discard in
`boards/nxp/mr-tropic/nuttx-config/scripts/script.ld` or the link fails with an
`R_ARM_PREL31` relocation error. Both steps, and why, are in
[PX4_INTEGRATION.md](PX4_INTEGRATION.md#imxrt-boards-only-r_arm_prel31-link-error).
These changes live in the **PX4 tree**, not this repo.

### 4b. Build and upload

```sh
cd /path/to/PX4-Autopilot
make nxp_mr-tropic_default upload
```

`upload` flashes over USB through the PX4 bootloader (the Tropic on
`/dev/cu.usbmodem01`). Confirm the dialect generated by checking that
`build/nxp_mr-tropic_default/mavlink/military/mavlink_msg_esad_*.h` exist.

## 5. Configure PX4 TELEM2

TELEM2 must be a MAVLink instance that forwards to/from the USB/GCS link and
matches the Nucleo's 57600 8N1, no flow control. Open the PX4 MAVLink shell and
paste these in order (also in [`host/px4_telem2_setup.txt`](../host/px4_telem2_setup.txt)):

```sh
Tools/mavlink_shell.py /dev/cu.usbmodem01
```

```
param set MAV_1_CONFIG 102     # 102 = TELEM2 -> /dev/ttyS4
param set SER_TEL2_BAUD 57600  # match the Nucleo
param set MAV_1_MODE 2         # Onboard
param set MAV_1_FORWARD 1      # REQUIRED: forwarding is OFF by default
param set MAV_1_FLOW_CTRL 0    # force flow control OFF (3-wire link)
param set MAV_1_RATE 0         # auto rate
param save
reboot
```

Notes:

- Instance **1** is TELEM2. Do not touch `MAV_0_*` (that is TELEM1). The USB/GCS
  instance is automatic; you do not configure it here.
- On mr-tropic, TELEM2 is `/dev/ttyS4`. Its `rc.board_mavlink` is an empty stub,
  so there is no conflict with these params.
- `MAV_1_FORWARD=1` is not optional. PX4 does not forward between links by
  default; without it the ESAD traffic never crosses from TELEM2 to USB.
- **INT32 param gotcha:** if you set these over MAVLink `PARAM_SET` from a script
  rather than the shell, PX4's INT32 params require the integer's *bit pattern*
  packed into the float `param_value` field. Sending `float(102)` stores garbage.
  See [TROUBLESHOOTING.md](TROUBLESHOOTING.md#int32-param_set-stores-garbage).
  The shell `param set` command above does this correctly, so prefer it.

A reboot is required for the config changes to take effect.

## 6. Run the full loop

Wire TELEM2 to the Nucleo per [WIRING.md](WIRING.md), then:

```sh
cd host
./generate_dialect.sh          # if not already done
python3 gcs_via_px4.py --arm 1
```

The tail should read:

```
PASS: full loop Mac -> PX4 -> Nucleo -> PX4 -> Mac over the military dialect.
```

If it does not, work through [TROUBLESHOOTING.md](TROUBLESHOOTING.md).
